# Design: extension-system

## 1. Directory Structure

```
~/.config/siyuan-cli/
├── config.yaml
└── extensions/
    ├── .gitignore          # node_modules/, *.schema.json
    ├── tsconfig.json       # generated by `siyuan extension init`
    ├── apis/
    │   ├── my-endpoint.ts
    │   └── my-endpoint.schema.json   # auto-generated cache
    └── tools/
        ├── my-tool.ts
        └── my-tool.schema.json       # auto-generated cache
```

Future slot (not implemented): project-local extensions at `.siyuan-cli/extensions/`.

## 2. Extension Contract

### Tool extension

```ts
// ~/.config/siyuan-cli/extensions/tools/my-tool.ts
import type { ToolSchema } from '@frostime/siyuan-cli/schema';

export const tool: ToolSchema = {
    id: 'my-tool',
    summary: 'Does something useful',
    tags: ['read', 'util'],
    input: {
        type: 'object',
        properties: {
            query: { type: 'string', description: 'Search query' }
        },
        required: ['query']
    },
    async run(ctx, input) {
        const { query } = input as { query: string };
        const results = await ctx.callEndpoint('query.sql', {
            stmt: `SELECT * FROM blocks WHERE content LIKE '%${query}%'`
        });
        return { content: JSON.stringify(results) };
    }
};
```

### API endpoint extension

```ts
// ~/.config/siyuan-cli/extensions/apis/custom-query.ts
import type { EndpointSchema } from '@frostime/siyuan-cli/schema';

export const schema: EndpointSchema = {
    endpoint: '/api/query/sql',  // wraps existing kernel API
    summary: 'Custom query wrapper',
    payload: {
        type: 'object',
        properties: { stmt: { type: 'string' } },
        required: ['stmt']
    },
    classification: { mode: 'read', surface: 'content', scope: 'global' }
};
```

**Naming rule**: Extension file name is informational only. The authoritative identity is `tool.id` / `schema.endpoint`.

## 3. Schema Cache

### 3.1 Purpose

Separate discovery (list/help) from execution (jiti import). Discovery reads only JSON — no jiti runtime loaded.

### 3.2 Versioned cache format

All schema.json files carry `_version` for forward-compatible migration:

```ts
const CACHE_VERSION = 1;  // bump on structural changes

interface SchemaCacheEnvelope<T> {
    _version: number;          // CACHE_VERSION at write time
    _meta: {
        sourceFile: string;    // relative path of .ts/.mjs
        cachedAt: string;      // ISO timestamp
    };
    data: T;                   // ToolSchemaCache | EndpointSchemaCache
}
```

On load: `_version < CACHE_VERSION` → treat as stale, trigger re-import on next execution.

### 3.3 Cache content — Tool

```ts
interface ToolSchemaCache {
    id: string;
    summary: string;
    description?: string;
    tags?: string[];
    input: JSONSchema;        // for --help flag generation
    output?: JSONSchemaProperty;
    cli?: CliBehavior;        // examples, aliases
}
```

### 3.4 Cache content — Endpoint

```ts
interface EndpointSchemaCache {
    endpoint: string;
    summary: string;
    description?: string;
    payload: JSONSchema;
    classification: EndpointClassification;
    guard?: {
        payloadTargets?: PayloadTargetSpec[];
        response?: { itemsAt: string; fieldMap: Record<string, string> };
        // filterResponse is a function — not cacheable, omitted
    };
    cli?: CliBehavior;
    formatStrategy?: FormatStrategy;
    // format() is a function — not cacheable, omitted
}
```

### 3.5 Cache strategy

**Design principle**: Single write point (execution piggyback). Discovery paths are read-only with graceful degradation.

| Layer | Trigger | Behavior | Write? |
|-------|---------|----------|--------|
| **Piggyback** | `tool run` / `api run` | jiti import already done → write schema.json post-execution | ✅ |
| **Explicit** | `siyuan extension cache` | batch jiti import all → write all schema.json | ✅ |
| **Discovery** | `list` / `--help` | read schema.json if exists → show data or [uncached] | ❌ read-only |

**Rationale for read-only discovery**:

- Single write point → one place to handle write failures, no combinatorial complexity
- Discovery paths stay pure (no "list wrote files" surprise, no write-permission errors)
- Natural usage flow (write extension → test it → cache populated → list works) covers 90% of cases
- Explicit `siyuan extension cache` covers the rest

### 3.6 Staleness detection

```
load schema.json
  ├── not found            → status: uncached
  ├── _version < CURRENT   → status: stale (schema migration)
  └── found + version OK   → compare mtime(.ts) vs mtime(.schema.json)
       ├── .ts newer       → status: stale
       └── .ts older/equal → status: cached ✓
```

Stale cache: still usable for list/help (shows last known data), but flagged.
Execution path: stale → re-import via jiti → write fresh cache.

### 3.7 Cache write — error handling

```ts
function writeSchemaCache(source: string, schema: ToolSchema | EndpointSchema): void {
    try {
        const cachePath = source.replace(/\.(ts|mjs)$/, '.schema.json');
        const envelope: SchemaCacheEnvelope<...> = {
            _version: CACHE_VERSION,
            _meta: { sourceFile: basename(source), cachedAt: new Date().toISOString() },
            data: extractCacheData(schema)
        };
        writeFileSync(cachePath, JSON.stringify(envelope, null, 2));
    } catch (err) {
        // Non-fatal: warn and continue
        console.warn(`[ext] Failed to write cache for "${source}": ${err}`);
    }
}
```

Write failure never blocks execution.

## 4. Extension Loader

### 4.1 Interface

```ts
// src/extension/loader.ts

type CacheStatus = 'cached' | 'stale' | 'uncached';

interface DiscoveredExtension<T> {
    source: string;             // absolute path to .ts/.mjs
    cacheStatus: CacheStatus;
    cached?: T;                 // from schema.json (if available)
}

interface LoadedExtension<T> {
    source: string;
    schema: T;                  // full schema from jiti import
}

// Discovery: read cache only, no jiti
function discoverToolExtensions(extDir: string): DiscoveredExtension<ToolSchemaCache>[];
function discoverEndpointExtensions(extDir: string): DiscoveredExtension<EndpointSchemaCache>[];

// Full load: jiti import + cache write
async function loadToolExtension(source: string): Promise<LoadedExtension<ToolSchema>>;
async function loadEndpointExtension(source: string): Promise<LoadedExtension<EndpointSchema>>;

// Batch full load (for `extension cache` command)
async function loadAllToolExtensions(extDir: string): Promise<LoadedExtension<ToolSchema>[]>;
async function loadAllEndpointExtensions(extDir: string): Promise<LoadedExtension<EndpointSchema>[]>;
```

### 4.2 Execution path flow

```
siyuan tool <id> [args]
  → discoverToolExtensions(extDir)                    // scan + read cache
  → register all cached extensions (cache-only)       // for permission checks
  → is <id> a user extension?
    ├── YES → loadToolExtension(source)               // jiti import
    │         → registerExtension(tool)                // replace cache-only entry
    │         → tool.run(ctx, input)
    │         → writeSchemaCache(source, tool)         // piggyback
    └── NO  → normal builtin flow
```

### 4.3 Discovery path flow

```
siyuan tool list
  → discoverToolExtensions(extDir)
  → for each:
      ├── cached/stale → show id + summary from cache
      └── uncached     → show id + "[uncached]"
  → if any uncached:
      stderr: "[!] N uncached extension(s). Run `siyuan extension cache` to populate metadata."
```

### 4.4 File scanning rules

```
scan directory:
  include: *.ts, *.mjs
  exclude: *.d.ts, *.schema.json, *.test.ts, node_modules/
```

### 4.5 Export validation

```ts
function validateToolExport(mod: unknown, source: string): ToolSchema {
    // mod must have: mod.tool
    // mod.tool must have: .id (string), .run (function), .input (object)
    // Failure: console.warn + skip
}

function validateEndpointExport(mod: unknown, source: string): EndpointSchema {
    // mod must have: mod.schema
    // mod.schema must have: .endpoint (string), .classification (object), .payload (object)
    // Failure: console.warn + skip
}
```

## 5. Registry Integration

### Conflict handling

```ts
// Added to both EndpointRegistry and ToolRegistry:

registerExtension(schema: ToolSchema | EndpointSchema): boolean {
    const id = /* derive id */;
    if (this.map.has(id)) {
        console.warn(`[ext] Skipping extension "${id}": conflicts with builtin`);
        return false;  // skip, do not throw
    }
    // normal registration path
    this.map.set(id, /* ... */);
    return true;
}
```

Key difference from `register()`: conflict → warn + skip (returns false) instead of throw.

## 6. Lazy Loading Hook

Extensions load **only when api/tool subcommands are invoked**, not at CLI startup.

```ts
// src/api/command.ts — added before endpoint dispatch:
let extensionsLoaded = false;
async function ensureExtensions() {
    if (extensionsLoaded) return;
    extensionsLoaded = true;
    await loadAndRegisterExtensions('apis');
}

// src/tool/command.ts — same pattern:
let extensionsLoaded = false;
async function ensureExtensions() {
    if (extensionsLoaded) return;
    extensionsLoaded = true;
    await loadAndRegisterExtensions('tools');
}
```

Other subcommands (`workspace`, `doc`, `skill`, `approval`) unaffected — zero overhead.

## 7. `siyuan extension` Command

```
siyuan extension init    # scaffold extensions/ directory
siyuan extension list    # show all discovered extensions + cache status
siyuan extension cache   # batch jiti-import + write all schema.json
```

## 8. Package Type Exports

### Current state

`package.json` has no `exports` or `types` field. tsdown generates `.d.mts` files in `dist/` but they're not exposed.

### Target state

```jsonc
{
    "main": "./dist/cli.mjs",
    "types": "./dist/cli.d.mts",
    "exports": {
        ".": {
            "types": "./dist/cli.d.mts",
            "import": "./dist/cli.mjs"
        },
        "./schema": {
            "types": "./dist/shared/schema.d.mts",
            "import": "./dist/shared/schema.mjs"
        }
    }
}
```

Extension authors use:

```ts
import type { ToolSchema, EndpointSchema } from '@frostime/siyuan-cli/schema';
```

The `/schema` subpath keeps the public API surface minimal — only type definitions needed for authoring extensions.
